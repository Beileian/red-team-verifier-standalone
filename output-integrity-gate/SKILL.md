# 输出完整性门禁 (Output Integrity Gate)

## 问题

LLM 幻觉有两个致命特征：① 输出时看起来自洽（数字本身没毛病，只是归属错了实体）；② 只有人工交叉验证才会暴露。2026-06-28 实例：万华化学（601309，52周区间 52-95.44）买入建议写成"36元以下"——36是招商银行的价格，在同一会话的上下文窗口里串到了万华的输出中。用户不主动查股价，这个错误就永远留下了。

## 适用范围

**所有分析类输出，不限领域。** 股票分析、技术报告、数据解读、对比分析——只要输出含数字或事实断言，就加载。

## 门禁清单（输出前执行，7项）

```
□ 1. 数字-实体绑定：输出中每个数字是否属于正确的实体？
     → 检查：如果同一会话有多个实体（多只股票/多个项目/多个产品），
       输出前逐条对照：该数字是从哪个实体取来的？
     → 反例：万华化学建议"36元买入"，但36元是招商银行的

□ 2. 范围合理性：每个数值是否在常识/已知范围内？
     → 检查：股价∈52周区间？市值∈同行业合理范围？
       增长率∈[-100%, +∞]？日期∈当前时间线？
     → 反例："腾讯市值300亿"（实际5万亿）/"2025年Q5财报"（不存在）

□ 3. 关键断言可追溯：输出中每个事实性断言，在当前对话中能否找到来源？
     → 检查：逐条回溯——这个数字从哪个工具调用来的？
       哪个搜索结果？哪个用户输入？
     → 反例：说"机构一致看多"但未实际搜索过研报

□ 4. 内部自洽：同一输出内的数字是否互相矛盾？
     → 检查：PE × EPS ≈ 股价？营收增速与利润增速方向一致？
       不同段落同一数据值相同？
     → 反例：前文说PE=17x、EPS=4.2元，后文说股价=36元（17×4.2=71.4≠36）

□ 5. 格式标注完整：所有数字是否标注了来源类型和as-of？
     → [年报一手]/[同花顺]/[券商预测 as-of:日期]/[as-of:日期 收盘]
     → 反例：无标注的裸数字（无法追溯验证）

□ 6. 标识符完整性：输出中引用的 ID/路径/名称是否被截断？
     → 检查：KB ID 以 `=` 结尾？文件名有扩展名？路径不以 `...` 结尾？
       任何从终端输出提取的长字符串——视觉截断极易漏过
     → 反例：KB ID `YpW7Z...Gmf_Rk=` 被截成 `YpW7Z...Rk`（缺尾=），搜索报220004
     → 触发条件：引用了 search_knowledge_base / terminal / API 返回的长标识符
     → 验证：截断的 ID 后端 API 一定报错（220004/404），不会静默失败

□ 7. 输出整洁度：对用户可见的输出是否干净？不含内部过程？
     → 检查：无 raw diff 输出、无 git push 裸文本、无 search JSON、
       无 tool call 中间结果、无 patch 成功/失败的技术细节、无 confirmation token
     → 特别注意：无英文短句叙事（"Now bump version and push"、"Let me check"、
       "Done, patching..."）。纯中文输出——英文专名可保留但仅作名词，不叙事
     → 例外：版本号（v1.5.0）和 commit hash（c37b21b）可保留（用户明确要求）
     → 反例：展示 "diff --git a/... b/..." 或 "To https://github.com/..." 或
       "exit_code: 0" 或 "Now bump version and push"
     → 规则：成功时一句中文汇总，失败时中文说清原因。输出质量 > token 节省
     → 严重性：同日多次违规（2026-06-28），已升级为硬拦截
```

## § 分级触发机制（v1.3.0 新增）

自检完成后，按输出风险等级决定是否唤起 red-team-verifier 子代理：

### 风险分级

| 级别 | 判定条件 | 示例 |
|------|---------|------|
| **P0** | 含具体价位/买卖建议/**财务数字断言**/交易策略建议 | "建仓价36元"、"目标价7.80"、"PE 7.8x"、"股息率3.95%" |
| **P1** | 含事实性判断但**无交易建议** | 行业趋势分析、估值框架讨论、方法论评估 |
| **P2** | 纯代码修改/文档更新/行政操作/配置变更 | patch、write_file、cron管理、分类器修复 |

### 触发规则

| 级别 | 子代理 | 阻塞? | 说明 |
|------|--------|:---:|------|
| **P0** | **强制唤醒** | **阻塞** | 先唤子代理独立审查，通过后才交付用户。不交付未审查的P0输出 |
| **P1** | 门禁自检分 < 7 时唤醒 | 异步 | 先交付用户，子代理后台跑。发现问题后追更正 |
| **P2** | 不唤醒 | — | 无数字/事实断言风险，跳过子代理 |

### P0 阻塞流程

```
主进程完成分析输出
  → 运行本 gate 自检（7项）
  → 判定为 P0（含交易建议/财务数字断言）
  → delegate_task(
        goal="独立验证以下股票分析输出",
        context="[完整分析输出markdown] + [as-of数据源]",
        skills=["red-team-verifier"]
      )
  → 子代理返回验证结果
  → CRITICAL/MAJOR → 修正后重新交付
  → MINOR/无问题 → 交付原版
```

**成本**：每次 P0 约 2000-3000 token（子代理独立上下文），当前 deepseek-v4-pro 约 ¥0.005/次。

### P1 异步流程

```
主进程完成分析输出
  → 运行本 gate 自检 → 得分 < 7
  → 判定为 P1
  → 先交付用户
  → 后台 delegate_task 唤子代理审查
  → 发现问题后追推更正
```

### P0 判定关键词

触发 P0 的输出特征（满足任一即 P0）：
- 含 `¥` / `元` / `买入` / `卖出` / `加仓` / `减仓` / `建仓` / `止损` / `止盈`
- 含 PE / PB / EPS / ROE / 股息率 等估值指标的具体数值
- 含"目标价"、"支撑位"、"阻力位"、"建仓线"
- 含具体的百分比变动断言（"营收+10.5%"、"净利润-22.79%"）
- 含券商评级/机构目标价/一致预期的引用

### 例外

如果主进程已无 token 余量（如接近 max_tokens 上限），P0 阻塞降级为 P1 异步——优先交付，不因审查超时导致输出丢失。此时在输出末尾标注 `[P0降级P1：token余量不足，子代理异步审查中]`。

## 门禁不通过的处理

| 不通过项 | 处理 |
|---------|------|
| 任一数字实体绑定错误 | **硬拦截** — 修正后重新输出，不交付错误版本 |
| 范围异常（如股价 < 52周低点） | **硬拦截** — 99%是跨实体串扰，立即自纠 |
| 断言无来源 | **软警告** — 标注"[推断]"或补充搜索验证 |
| 内部矛盾 | **硬拦截** — 重新计算修正 |
| 格式标注缺失 | **软警告** — 补充标注后再交付 |
| 标识符截断（KB ID/路径/文件名） | **硬拦截** — 从原始 JSON 补全后再引用。截断 ID 后端 API 一定报错，不会静默失败 |
| 输出含中间过程（diff/git/push/search JSON） | **硬拦截** — 删除所有技术细节，只留一句汇总。同日第二次违规升级为硬拦截 |

## 与 serenity-router 的关系

serenity-router 的第+1步（跨股票价格一致性检查）是本 gate 在股票领域的特化实例。本 gate 是通用层——数字-实体绑定、范围合理性、断言可追溯适用于任何领域。

股票分析时，本 gate 与 serenity-router 的数据门禁并行执行。非股票分析时，本 gate 独立工作。

## ⚠️ 自审计陷阱：防御系统 ≠ 运行中

**设计但不触发 = 比没有更危险**（制造虚假安全感）。2026-06-29 实例：
三层防御体系描述为"output-integrity-gate → red-team-verifier → serenity-router"，
但检查后发现 red-team-verifier 从未被 delegate_task 实际调用——所有分析输出
都是主进程直接交付，子代理审查从未上场。

**自检规则**（每次股票分析后自查）：
1. 本次输出的风险等级是什么（P0/P1/P2）？
2. 如果 P0，delegate_task 是否已发出？
3. 如果 P1 + 自检<7，delegate_task 是否已异步发出？
4. 如果以上应发未发 → 这是 bug，先补发再继续

**节律**：每完成 3 次 P0 输出后，翻 session 记录验证 delegate_task 实际调用次数
是否匹配。

## 新增门禁项：批量分析规则一致性（v1.5.0）

□ 9. 批量分析一致性：同一规则是否对分析中所有标的均等应用？
     → 检查：遍历所有分析标的，每条自定义规则（股息计算/估值框架/红旗扫描）
       在每个标的上是否一致执行？不可仅在第一个标的上做了检查而其余跳过
     → **反例（2026-07-02）**：四股股息率分析，招商银行正确应用了"中期已除权不计入"
       双轨去除，但国电电力/农业银行/中国银行未应用——导致三只股票股息率
       系统性高估71%-∞，收息建仓优先级完全颠倒
     → **根因**：人类倾向——第一个复杂案例消耗认知资源后，后续简单案例被"默认通过"
     → **防御**：输出前对每个标的逐条复验关键计算步骤，不只验第一个

## 新增门禁项：MACD 窗口不足（v1.6.0）

□ 11. MACD DEA 收敛检查：K 线数据是否 ≥60 日？
     → 检查：EMA26 需要约 26×3=78 个数据点才能稳定收敛。35 日窗口 DEA 仅 2-4 个有效值，
       BAR 符号可能反转，DIF 方向为准。
     → 反例：国电电力 35 日窗口 MACD BAR 复算为 +0.0043 但报告为 -0.0155（符号反转）
     → 防御：K 线 < 60 日 → MACD 标注"DEA 未收敛，DIF 方向为准"
     → 来源：2026-07-06 国电电力三合一分析 red-team 复核

## 新增门禁项：财务数据口径（v1.6.0）

□ 12. 净利润口径检查：新浪"净利润"是否为集团总口径？
     → 检查：新浪 API 返回的「净利润」= 含少数股东损益的集团总额，非归母。
       PE×EPS 自洽前必须确认为归母口径。
     → 反例：国电电力新浪"净利润"137.27亿 vs 归母 71.61亿，偏差 +91.7%
     → 防御：新浪净利润与东财归母净利交叉验证。偏差 >10% → 修正
     → 来源：2026-07-06 国电电力/农业银行/中国银行三合一分析

□ 13. PE×EPS 周期匹配：PE 类型与 EPS 区间是否一致？
     → 检查：PE(TTM) 只能 × EPS(TTM)，PE(静) 只能 × EPS(FY 全年)
     → 反例：农业银行 PE(TTM) × FY2025 全年 EPS = 4.97 vs 股价 5.89（偏差 9.3%）
     → 防御：输出前逐条确认 PE 类型 → 对应 EPS 区间
     → 来源：2026-07-06 农业银行三合一分析

□ 14. PE(静) 双源交叉：腾讯字段 52 是否与东财一致？
     → 检查：腾讯 qt.gtimg.cn 的 PE(静) 与东财数据常见偏差（7-9%）
     → 反例：中国银行腾讯 7.91 vs 东财 7.37
     → 防御：PE(静) 输出前以东财 web_search 交叉验证，不一致以东财为准
     → 来源：2026-07-06 中国银行/农业银行三合一分析

## 与 serenity-router / red-team-verifier 的关系

```
output-integrity-gate  = 自检清单（主进程输出前自查，7项，~30秒）
                         + 风险分级（P0/P1/P2） ← v1.3.0 新增
                         + P0→阻塞唤子代理，P1→条件异步唤
red-team-verifier      = 独立审查（子代理独立验证，不共享上下文，7项FT-CODE）
                         按分级触发——P0阻塞/P1异步/P2跳过
serenity-router        = 股票分析专项（数据门禁 → 方法论闸门 → 方法论层 → rubrics自评）
```

三者互补：
- gate 抓显性错误（数字范围/标注缺失/实体绑定/ID截断/输出整洁度），输出前快速自检 + 判定风险等级
- verifier 抓隐性错误（推理断裂/内部矛盾/跨实体串扰），P0阻塞独立审查，P1条件异步
- **serenity-router 第0.5步方法论闸门防"门禁过了=分析完了"的错觉**（2026-06-28三花智控教训：门禁满分但方法轮未加载）
- 股票分析时，gate 和 verifier 都在 serenity-router 的框架内触发
