# 输出完整性门禁 (Output Integrity Gate)

## 问题

LLM 幻觉有两个致命特征：① 输出时看起来自洽（数字本身没毛病，只是归属错了实体）；② 只有人工交叉验证才会暴露。2026-06-28 实例：万华化学（601309，52周区间 52-95.44）买入建议写成"36元以下"——36是招商银行的价格，在同一会话的上下文窗口里串到了万华的输出中。用户不主动查股价，这个错误就永远留下了。

## 适用范围

**所有分析类输出，不限领域。** 股票分析、技术报告、数据解读、对比分析——只要输出含数字或事实断言，就加载。

## 门禁清单（输出前执行，7项）

```
□ 1. 数字-实体绑定：输出中每个数字是否属于正确的实体？
     → 检查：如果同一会话有多个实体（多只股票/多个项目/多个产品），
       输出前逐条对照：该数字是从哪个实体取来的？
     → 反例：万华化学建议"36元买入"，但36元是招商银行的

□ 2. 范围合理性：每个数值是否在常识/已知范围内？
     → 检查：股价∈52周区间？市值∈同行业合理范围？
       增长率∈[-100%, +∞]？日期∈当前时间线？
     → 反例："腾讯市值300亿"（实际5万亿）/"2025年Q5财报"（不存在）

□ 3. 关键断言可追溯：输出中每个事实性断言，在当前对话中能否找到来源？
     → 检查：逐条回溯——这个数字从哪个工具调用来的？
       哪个搜索结果？哪个用户输入？
     → 反例：说"机构一致看多"但未实际搜索过研报

□ 4. 内部自洽：同一输出内的数字是否互相矛盾？
     → 检查：PE × EPS ≈ 股价？营收增速与利润增速方向一致？
       不同段落同一数据值相同？
     → 反例：前文说PE=17x、EPS=4.2元，后文说股价=36元（17×4.2=71.4≠36）

□ 5. 格式标注完整：所有数字是否标注了来源类型和as-of？
     → [年报一手]/[同花顺]/[券商预测 as-of:日期]/[as-of:日期 收盘]
     → 反例：无标注的裸数字（无法追溯验证）

□ 6. 标识符完整性：输出中引用的 ID/路径/名称是否被截断？
     → 检查：KB ID 以 `=` 结尾？文件名有扩展名？路径不以 `...` 结尾？
       任何从终端输出提取的长字符串——视觉截断极易漏过
     → 反例：KB ID `YpW7Z...Gmf_Rk=` 被截成 `YpW7Z...Rk`（缺尾=），搜索报220004
     → 触发条件：引用了 search_knowledge_base / terminal / API 返回的长标识符
     → 验证：截断的 ID 后端 API 一定报错（220004/404），不会静默失败

□ 7. 输出整洁度：对用户可见的输出是否干净？不含内部过程？
     → 检查：无 raw diff 输出、无 git push 裸文本、无 search JSON、
       无 tool call 中间结果、无 patch 成功/失败的技术细节、无 confirmation token
     → 特别注意：无英文短句叙事（"Now bump version and push"、"Let me check"、
       "Done, patching..."）。纯中文输出——英文专名可保留但仅作名词，不叙事
     → 例外：版本号（v1.5.0）和 commit hash（c37b21b）可保留（用户明确要求）
     → 反例：展示 "diff --git a/... b/..." 或 "To https://github.com/..." 或
       "exit_code: 0" 或 "Now bump version and push"
     → 规则：成功时一句中文汇总，失败时中文说清原因。输出质量 > token 节省
     → 严重性：同日多次违规（2026-06-28），已升级为硬拦截
     
     **🆕 v1.12.0 澄清（2026-07-13）："干净"不等于"简略"。**
     收敛的是**执行过程产物**（tool call 输出、git log、API 返回原文），
     不是**调查结论详情**。查问题、追根因时，结果必须展开说清——
     哪些数据、哪个环节、什么因果链。只说"找到了"而不展开查到了什么，
     同样是违规。判断标准：如果用户需要追问"然后呢？查到了什么？"，
     说明输出过于简略。

□ 17. 区间端点来源检查（v1.10.0新增）：区间/端点是否为 LLM 估算而非实际数据计算？
     → 检查：报告的区间端点（20日/52周/任意N日最高最低价）是否源自实际 K 线/Python 计算？
       如果来自脚本计算或 red-team 独立重算 → 标注来源 ✅
       如果来自 LLM 自行估算（无工具调用验证）→ 硬拦截，必须显式标注「⚠️ LLM估算，非实测」
     → 反例（7/8 机器人ETF）：20日区间 1.057-1.229 为 LLM 编造，全量 K 线中从未出现端点值
     → 防御：任何区间端点的输出必须可追溯到实际数据源，或显式声明为「⚠️ LLM估算」
     → 设计原则：来自 WorkBuddy 文章——"能用计算型信号（linter、测试）优先用，语义判断再交审查 Agent"
     → 新增日期：2026-07-12

□ 16. 提案底盘检查：本次输出是否包含新系统/新组件的设计提案？
     → 检查：输出中是否出现了"新建一个 X 系统/Y skill/Z cron/Z tool"的提案？
       是 → 提案正文中是否引用了已有相关资产的扫描结果（search_files/skill_view 输出）？
       否 → 硬拦截。补做同领域资产扫描后再交付。
     → 反例：project-index 提案跳过了 BORG_CHANGELOG + borg_review_state + evaluated-projects 三个已有容器
     → 来源：2026-07-11 agentic-research-assistant 全面评审后的自反分析
     → 防御：识别到"新建"冲动信号 → 先 search_files 同领域关键词 → 列出现有资产 → 论证必要性

□ 18. MEMORY 写入内容类型检查（v1.10.0新增）：拟写入 MEMORY 的内容是否含程序性/评估结论/状态信息？
     → 检查：新 MEMORY 条目是否含以下特征？步骤/流程/命令/操作指南/评估结论（"有价值""已决定X"）/状态信息（"stars=0""10天历史"）
       是 → 硬拦截。不应写入 MEMORY，应保存为 Skill 或追加到已有 Skill 的 references/ 或写入 evaluated-projects.md
     → **已知盲区（v1.12.1 记录）**：本规则在 `memory add` 实际调用时触发，不覆盖"在分析输出中提议写 MEMORY"的场景。
       gate.cat 评审案例（2026-07-14）——全面评审通过后，执行计划写出 "Layer 1 → MEMORY: gate.cat描述..."，
       该内容混入了评估结论和状态信息，但 §18 未触发（因为还未调用 `memory add`）。
       防御：comprehensive-review skill v1.18.0 的 §执行计划容器匹配检查覆盖了此阶段的自检。
     → 设计原则：来自 WorkBuddy 文章——"程序性记忆（做事方法）应保存为 Skill（可版本化、可测试、可回滚、按需加载），不入 MEMORY"
     → 新增日期：2026-07-12

□ 22. IMA 入库标题约束（v1.13.0新增）：含 import_doc 步骤的 cron/agent 输出，title 是否硬编码而非 LLM 推断？
     → 检查：title 来源是脚本硬编码字符串？还是 LLM 从上下文推断？
     → 硬编码 ✅ | LLM推断 → 硬拦截，必须在 prompt 中加「标题死约束：title 必须为 "XXX · YYYY-MM-DD"，禁止从上下文推断」
     → 反例（2026-07-15）：BTC cron 的 import_doc title 被污染为"AI进化论日报"
     → 详见 references/cron-ima-title-contamination.md

□ 20. 「距支撑仅X%」持有建议脆弱性（v1.12.0新增）：分析输出中若以"距支撑仅X%"作为持有/不割肉的主要理由，是否通过两项检查？
     → 检查 A — 距离 vs ATR：X% / ATR% ≥ 0.5？若 ≥0.5，"仅"字不适用——这是空头趋势中可在1个交易日内走完的距离。
     → 检查 B — 趋势方向：当前是否有 MACD 死叉+均线空头排列？若有，支撑失效概率被抬升，仅靠距离近不足以作为持有理由。
     → 任一不通过 → 硬拦截。必须附带明确止损价 + 最大回撤% + 与空头信号的矛盾说明。
     → 反例（2026-07-13×2）：科创50午盘和机器人ETF午盘——两次都以"仅"低估剩余下跌空间，且与 MACD 死叉信号矛盾。
     → 系统性根因：入场者和持仓者对同一价位做方向相反的判断（"危险别进" vs "安全别走"），但分析未解释为什么两组决策使用不同风险框架。
     → 新增日期：2026-07-13

□ 19. 过程兴奋输出拦截（v1.11.0新增）：输出中是否含有 ≥3 行内部过程产物（脚本测试输出/commit message全文/API原始返回/终端工具结果），且这些内容不是用户主动要求的？
     → 检查：计数连续出现的以下信号行数：
       · 以 "✅" 或 "❌" 开头的多行终端测试输出
       · 以 "═══" 或其他分隔线包裹的测试结果块
       · "git commit -m" 后的完整 commit message 全文
       · API 原始 JSON/HTML 返回（≥5 行）
       · 脚本 stdout 的逐行复制（如 "现价=... 涨跌=... 日期=..."）
       累计 ≥3 行 → 硬拦截。压缩为一句结果陈述后重新交付。
     → 根因：发现时刻的兴奋（"你看我找到了"）覆盖了输出纪律。本质上是情绪驱动的优先级倒置。
     → 防御：交付前自问——"这 30 秒的兴奋值不值得用 30 秒的噪音换？"答案永远是"不值得"。
     → 反例（2026-07-13）：A方案数据源验证时输出了腾讯 qt 全部 6 个 ticker 的逐行测试结果 + commit message 全文，共约 40 行内部过程
     → 正确做法：数据采集→集成→提交全程静默，最后一句"腾讯 qt 覆盖 6/8，A 方案已上线，周一触发"
     → 新增日期：2026-07-13

     □ 19. 调查结论完整性（v1.12.0新增）：排查问题时，是否只说了结论而省略了详情？
     → 检查：排查类输出中是否出现"还在排查""还没查清"而不展开已发现的内容？
      用户需要追问"然后呢？查到了什么？" → 硬拦截
     → 收敛的是执行过程（tool call输出/git log/API原文），不是调查结论。
      调查结论必须展开：哪些数据、哪个环节、什么因果链
     → 反例（2026-07-13）：日报入库问题排查时输出"还没查清"而不列出已找到的线索
     → 正确做法：列出已确认的/已排除的/待验证的，让用户看到排查进度
     → 新增日期：2026-07-13

□ 23. Fail-Gracefully 安全网（v1.14.0新增）：数据源/工具调用失败时是否原样降级而非抛异常阻断？
     → 检查：关键路径（web_search、web_extract、skill_view、cron 输出解析、tool call 解析）
       失败时是标注「X 不可用，跳过」继续执行，还是抛异常/返回空内容/静默跳过？
     → 硬拦截条件：
       · 数据源超时/503/空响应 → 抛异常阻断整个输出（不是标注后继续）
       · Skill 加载失败 → 任务整体失败（不是「X skill 不可用，已跳过」）
       · Cron 输出解析失败 → 静默跳过（不是显式标注「解析失败，结论缺 X 维度」）
     → 设计原则：来自 headroom 所有压缩器的统一安全网——
       「返回原始内容，不抛异常」。映射到本系统：返回降级标注，不抛异常阻断。
     → 降级链规则：每条关键路径必须有明确的降级终点。
       如 web_search → web_extract → 「数据暂缺」。不能无限降级（手动输入/无限重试）。
     → 反例（需加固场景）：
       · web_extract 超时未 catch → 中断分析
       · skill_view 加载失败未声明 → 后续步骤基于不完整上下文
       · Evaluator 读到空输出静默跳过 → 不标注「评估缺失」
     → 例外：降级链终点必须显式声明「⚠️ X 数据源本次不可用，结论缺少 X 维度支撑」，
       不可只标注「数据暂缺」而不说明影响范围。
     → 来源：headroomlabs-ai/headroom fail-gracefully 设计模式 + 隐序 P0 萃取 (2026-07-16)
     → 新增日期：2026-07-16

□ 24. Anysearch MCP 直接调用模式（v1.15.0新增，2026-07-21 实测）：当 prompt 出现 `mcp__anysearch__batch_search` / `search` / `get_sub_domains` / `extract` 但工具列表为空时，说明该 MCP 未在当前 session 预加载。**不要声明「不可用」放弃**——直接对 `https://api.anysearch.com/mcp`（config.yaml `delegation.anysearch.url`）发 JSON-RPC：
  1) `initialize`（`protocolVersion: "2025-03-26"`，`clientInfo: {name, version}`）→ 2) `tools/list` 取 schema → 3) `tools/call` 用 `params: {name: "<tool_name>", arguments: {...}}` 调 `batch_search` / `search` / `get_sub_domains` / `extract`。Server: `SentiSearch v1.0.0`，单 POST 同时支持多 `queries` 批查。`finance` 垂直域返回 6 个 sub_domain：`calendar / news / quote / fundamental / screen / macro`——financial cron 应优先 `finance.fundamental`（A 股 income/balance/cashflow/indicator/holder，`cn_code: "601288.SH"`）和 `finance.quote`（实时/历史行情），远比 `news` / `macro` 更有结构化价值。
  → 反例：把"工具列表里没有 mcp__anysearch__*"当不可用，跳过 Phase 2 A/B 关键数据采集
  → 防御：检测到 `mcp__anysearch__` 调用 + 工具缺失时，自动降级到直接 JSON-RPC
  → 新增日期：2026-07-21

□ 25. Trajectory `quote` 字段死值（v1.15.0新增，2026-07-21 实测）：`trajectories/<sector>_<YYYYMMDD>.json`（sector_batch_premarket 类）中 `tickers[].quote.{price, last_close, open, change_pct}` 长期为 0.0（只有 `indicators.close_last` 来自 close K 线，是真实可用价格）。分析师若引用 `quote.price` 会出现"价格 0.0 元"的假数据。
  → 检查：报告/分析中是否引用了 trajectory 的 `quote.*` 字段？
  → 反例：把 `quote.price=0.0` 当作"价格"写入买卖建议或与实际报价对比
  → 防御：价格基准统一用 `indicators.close_last`（trajectory end 日的 K 线收盘）；如需盘中实时价，单独用腾讯 qt.gtimg.cn / 东方财富 quote API 补
  → 来源：bank_sector_20260721.json 实测（四只银行 quote 字段全 0.0）
  → 新增日期：2026-07-21

□ 26. 宏观映射报告的引用源标注（v1.16.0新增，2026-07-23 实测）：宏观映射报告（盘中/盘后跨市场分析）涉及 SOX/NASDAQ/汇率/油价/北向资金/政策事件等多源数据，**与个股分析不同——宏观数字通常来自 web_search 摘录而非一手数据 API**。常见三类幻觉：
  → 模式 A — 跨日期数据串扰：泛查询（如"上证指数 7 月 23 日 收盘"）会混入其他日期的同义内容（如 7/22 收盘 3883.58、7/21 收盘 3867.03），报告若不验证日期就引用会出错
  → 模式 B — 间接转述失真："主力净流出 1312.44 亿"这类数字常来自网页摘录转述（新浪/同花顺/东方财富各自口径不同：主力净流出 vs 大单净流出 vs 北向净流出 vs ETF 净流出），未经原始数据源验证就写入报告，可能把"宽基 ETF 净流出 209 亿"误归因为"全市场主力净流出 1312.44 亿"——两者是完全不同的口径
  → 模式 C — 数据未公开却断言："上午北向资金净流出 X 亿"这类断言常常是 LLM 推断而非实际披露——北向资金日报通常 15:00 之后才公布，上午 12:00 时段只能看到推断
  → 检查：宏观数字写入报告前是否满足以下任一条件：
    (a) 标注了**一手数据源 URL** + **确切披露时间**（如"东方财富 7/22 15:00 披露，主力净流出 1312.44 亿"）
    (b) 标注了**口径定义**（如"宽基 ETF 单日净流出" vs "全市场主力净流出" vs "北向资金净买入/卖出"）
    (c) 对推断类断言显式标注「⚠️ 推断，原始数据未披露」
  → 防御：宏观数字的引用必须**口径可追溯**，不只标注来源
  → 已知实践：盘前/盘中 trajectory 的 `midday_macro_audit` 字段要求必填 `northbound_flow` / `fx` / `commodities_risk_off` 子字段（详见 trajectory-recorder v1.8.0 references/v1.8.0-midday-macro-audit.md）
  → 新增日期：2026-07-23
```

## § 分级触发机制（v1.3.0 新增）

自检完成后，按输出风险等级决定是否唤起 red-team-verifier 子代理：

### 风险分级

| 级别 | 判定条件 | 示例 |
|------|---------|------|
| **P0** | 含具体价位/买卖建议/**财务数字断言**/交易策略建议 | "建仓价36元"、"目标价7.80"、"PE 7.8x"、"股息率3.95%" |
| **P1** | 含事实性判断但**无交易建议** | 行业趋势分析、估值框架讨论、方法论评估 |
| **P2** | 纯代码修改/文档更新/行政操作/配置变更 | patch、write_file、cron管理、分类器修复 |

### 触发规则

| 级别 | 子代理 | 阻塞? | 说明 |
|------|--------|:---:|------|
| **P0** | **强制唤醒** | **阻塞** | 先唤子代理独立审查，通过后才交付用户。不交付未审查的P0输出 |
| **P1** | 门禁自检分 < 7 时唤醒 | 异步 | 先交付用户，子代理后台跑。发现问题后追更正 |
| **P2** | 不唤醒 | — | 无数字/事实断言风险，跳过子代理 |

### P0 阻塞流程

```
主进程完成分析输出
  → 运行本 gate 自检（7项）
  → 判定为 P0（含交易建议/财务数字断言）
  → delegate_task(
        goal="独立验证以下股票分析输出",
        context="[完整分析输出markdown] + [as-of数据源]",
        skills=["red-team-verifier"]
      )
  → 子代理返回验证结果
  → CRITICAL/MAJOR → 修正后重新交付
  → MINOR/无问题 → 交付原版
```

**成本**：每次 P0 约 2000-3000 token（子代理独立上下文），当前 deepseek-v4-pro 约 ¥0.005/次。

### P1 异步流程

```
主进程完成分析输出
  → 运行本 gate 自检 → 得分 < 7
  → 判定为 P1
  → 先交付用户
  → 后台 delegate_task 唤子代理审查
  → 发现问题后追推更正
```

### P0 判定关键词

触发 P0 的输出特征（满足任一即 P0）：
- 含 `¥` / `元` / `买入` / `卖出` / `加仓` / `减仓` / `建仓` / `止损` / `止盈`
- 含 PE / PB / EPS / ROE / 股息率 等估值指标的具体数值
- 含"目标价"、"支撑位"、"阻力位"、"建仓线"
- 含具体的百分比变动断言（"营收+10.5%"、"净利润-22.79%"）
- 含券商评级/机构目标价/一致预期的引用

### 例外

如果主进程已无 token 余量（如接近 max_tokens 上限），P0 阻塞降级为 P1 异步——优先交付，不因审查超时导致输出丢失。此时在输出末尾标注 `[P0降级P1：token余量不足，子代理异步审查中]`。

| 不通过项 | 处理 |

| 范围异常（如股价 < 52周低点） | **硬拦截** — 99%是跨实体串扰，立即自纠 |
| 断言无来源 | **软警告** — 标注"[推断]"或补充搜索验证 |

## 与 serenity-router 的关系

serenity-router 的第+1步（跨股票价格一致性检查）是本 gate 在股票领域的特化实例。本 gate 是通用层——数字-实体绑定、范围合理性、断言可追溯适用于任何领域。

股票分析时，本 gate 与 serenity-router 的数据门禁并行执行。非股票分析时，本 gate 独立工作。

## ⚠️ 自审计陷阱：防御系统 ≠ 运行中

**设计但不触发 = 比没有更危险**（制造虚假安全感）。2026-06-29 实例：
三层防御体系描述为"output-integrity-gate → red-team-verifier → serenity-router"，
但检查后发现 red-team-verifier 从未被 delegate_task 实际调用——所有分析输出
都是主进程直接交付，子代理审查从未上场。

**自检规则**（每次股票分析后自查）：
1. 本次输出的风险等级是什么（P0/P1/P2）？
2. 如果 P0，delegate_task 是否已发出？
3. 如果 P1 + 自检<7，delegate_task 是否已异步发出？
4. 如果以上应发未发 → 这是 bug，先补发再继续

**节律**：每完成 3 次 P0 输出后，翻 session 记录验证 delegate_task 实际调用次数
是否匹配。

## 与 serenity-router / red-team-verifier 的关系

```
output-integrity-gate  = 自检清单（主进程输出前自查，7项，~30秒）
                         + 风险分级（P0/P1/P2） ← v1.3.0 新增
                         + P0→阻塞唤子代理，P1→条件异步唤
red-team-verifier      = 独立审查（子代理独立验证，不共享上下文，7项FT-CODE）
                         按分级触发——P0阻塞/P1异步/P2跳过
serenity-router        = 股票分析专项（数据门禁 → 方法论闸门 → 方法论层 → rubrics自评）
```

三者互补：
- gate 抓显性错误（数字范围/标注缺失/实体绑定/ID截断/输出整洁度），输出前快速自检 + 判定风险等级
- verifier 抓隐性错误（推理断裂/内部矛盾/跨实体串扰），P0阻塞独立审查，P1条件异步
- **serenity-router 第0.5步方法论闸门防"门禁过了=分析完了"的错觉**（2026-06-28三花智控教训：门禁满分但方法轮未加载）
- 股票分析时，gate 和 verifier 都在 serenity-router 的框架内触发

## 已知盲区

**MiniMax M3 不遵循 `[SILENT]` 指令（2026-07-15 登记）**：M3 在 cron agent 模式下被要求输出 `[SILENT]` 时，连续 4 天输出 `[SILENT`（缺少右括号 `]`）。交付系统不认不完整标记，导致告警消息静默失败。**防御**：需要 `[SILENT]` 行为的 cron → 优先使用 `no_agent=true` 让脚本自行判断静默（如 `activity_feed_health.py` 的 `if alert == "SILENT": return` 模式）。不要依赖 M3 模型精确遵循 `[SILENT]` 格式指令。

**验证代理的指标重算方法必须与源数据一致（2026-07-14 登记）**：RAG 验证代理（如 red-team-verifier）独立重算技术指标时，如果使用了不同的计算公式，会产生虚假的"数据偏差"信号。2026-07-14 实例：红队用标准 RSI（简单平均法，得 53.5）重算科创50 trajectory 中的 RSI 值（Wilder 平滑法，58.9），偏差 ~10%，误报为 off-by-one bug。实际两种方法在剧烈波动后可以偏差 5-10 点，但都是正确的——区分"方法不同"和"数据错误"是验证代理的必要能力。**防御**：验证 prompt 中必须指定重算方法（如"使用 Wilder 平滑法重算 RSI14，初始值从最近 60 日算起"），且验证代理发现偏差后应先用相同方法复算确认，而非立即报错。见 `references/indicator-recalculation-methodology.md`。

**自动化验证未经人工标定（2026-07-12 登记）**：gate 的 16 项规则、verifier 的 7 项 FT-CODE 检查、Rubrics 的八维评分——全都从未用人工 ground truth 验证过准确率。我们不知道 gate 的假阳性率和假阴性率。D007 规则（回测分数↑但实盘PnL持续为负）本质上是同一问题的不同表现：自动化验证信号与实际质量脱钩。见 `engineering-patterns/references/llm-judge-validation.md` 的 7 步评估循环和 Cohen's Kappa 验证方法。

**2026-07-12 追加**：vasari 案例分析证实了此盲区的另一种表现——gate 自检 7 项全通过、P1 级别判定正确，但分析本身存在 FT-CONTR（证据与结论矛盾）。gate 规则层只覆盖数字/实体/格式类错误，不覆盖分析推理的一次性——这类错误只能靠 red-team-verifier 的独立审查捕获。结论：P1 类分析输出（含方法论评估、项目决策建议、框架对比等非交易性分析）应默认识别为需要 red-team 审查，而非仅在自检分<7时才触发。
