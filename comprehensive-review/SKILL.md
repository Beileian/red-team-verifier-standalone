# 统一评审框架 (Comprehensive Review)

## 触发条件

当用户要求对一份分析/报告/决策建议做**全方位质量评估**时加载。

触发词：**全面评审** / **Rubrics+红蓝** / **深度复盘** / **综合审查**

**文章链接场景**：当用户将上述触发词附加到文章链接请求时（如"全面评审这篇文章"），自动走完整流程——先独立分析文章（不加建议）→ 对照已有 skill 框架 → 红蓝互辩 → Rubrics 评分 → 收窄交付。不需要用户额外提醒"别忘了红蓝队"。

## 工作流

```
1. 加载 rubrics-model-evaluation skill（获取评分框架）
2. 加载 red-team-verifier skill（获取动态检测规则）
3. 🆕 若评审对象引用第三方开源项目 → 先执行「GitHub 三步验证」
   （references/cross-source-verification.md：Step1 Star/维护状态 → Step2 关键功能声明源码证据 → Step3 数据源/依赖声明）
4. 并行执行两路评估
5. 整合输出
```

## 输出格式

严格按以下四段结构输出：

---

### 📋 Rubrics 评分

| 维度 | 评分 | 验证 | 关键问题 |
|------|:--:|------|------|
| 事实正确性 | X | ✅2⚠️1❌1 | ... |
| 覆盖度 | X | — | ... |
| 证据支撑 | X | 引用N条/URL M条 | ... |
| 推理严谨性 | X | — | ... |
| 格式合规性 | X | — | ... |
| 来源类型 | X | 标注率X% | ... |
| 实际可用性 | X | — | ... |
| 意义提炼 | X | — | ... |

**加权总分：X.X/10** | 验证通过率：X/5

---

### 🔴 红队批判（3个致命疑点）

每条：疑点 + 为什么致命 + 反事实追问

---

### 🔵 蓝队建设（3个改进方向）

每条：方向 + 具体建议 + 优先级(P0/P1/P2)

---

### 🎯 整合结论

一句话总结 + 可信度评级：

| 评级 | 含义 |
|:--:|------|
| 🟢 A | 量化≥7.5 + 红队零致命疑点 → 可信 |
| 🟡 B | 量化5-7.4 或 红队1个致命疑点 → 可用但注意 |
| 🟠 C | 量化3-5 或 红队≥2致命疑点 → 需修正 |
| 🔴 D | 量化<3 或 红队发现事实错误 → 不可用 |

## § 执行计划容器匹配检查（v1.18.0）

**在分析类全面评审的整合结论中，如包含执行计划（如"Layer 1 → MEMORY / Layer 2 → X / Layer 3 → Y"），必须在输出前运行此容器匹配自检：**

```
□ 该内容是否含评估结论/状态信息（如"有价值""stars=0""已决定X"）？
  → 是 → 不透支 MEMORY。选 evaluated-projects 或跳过 Layer 1。

□ 该内容三月后是否仍成立？
  → 否 → 不选 MEMORY。MEMORY 是跨会话的持久事实存储，不是临时笔记。

□ 该内容删除后是否影响系统运行？
  → 否 → 不选 MEMORY。留 evaluated-projects 索引足矣，需要时 Ctrl+F。
```

**三个问题任一答"否"→ Layer 1 不选 MEMORY。默认归宿：evaluated-projects.md。**

> 此检查源自 2026-07-14 gate.cat 评审的两次容器匹配错误——全面评审分析了内容质量，但未审查执行计划的容器选择。详见已知陷阱「分析-执行断层」。

## § 审查者模型选择（v1.19.0）

**审查者不应比被审查者弱一级。** 当全面评审的评估对象是一个 LLM 生成的报告时，评估模型应与生成模型有认知差异：

| 被审查报告的生成模型 | 推荐审查模型 | 原因 |
|------|------|------|
| deepseek-v4-flash | deepseek-v4-pro | 同厂商不同架构强度，指令遵循和逻辑审查更强 |
| deepseek-v4-pro | deepseek-v4-flash 或 MiniMax-M3 | 不同认知模式（精确匹配 vs 跨域关联），正交性优先 |
| 任意单一模型 | 三路并行（Flash+Pro+M3） | dense/dense/MoE 三个正交维度，最高审查强度 |

**不推荐同模型自审**——生成者和审查者共享推理盲区、格式倾向和幻觉模式，审查结论可信度打折。

**成本考量**：单次切换 Pro(¥0.05) vs Flash(¥0.01) 的价差 ¥0.04，BTC 评估 3次/天 = ¥0.12/天。审查不可靠的隐性成本（错过质量信号）远超 ¥0.12。

> 此规则源自 2026-07-14 BTC Rubrics 评估模型讨论——被审查报告用 Flash 生成，审查者也是 Flash，16% 的 API 连接失败率叠加同质化审查风险。MiniMax 在结构化审计任务中落选（指令遵循弱于 Pro），但在三路并行中保留跨域知识审查的角色。

| | 单独 Rubrics | 单独红蓝 | **统一评审** |
|------|:--:|:--:|:--:|
| 评分 | ✅ | ❌ | ✅ + 验证 |
| 批判 | ❌ | ✅ | ✅ + 动态规则 |
| 互补 | — | — | ✅ 两路交叉验证 |
| 整合结论 | ❌ | ❌ | ✅ 可信度评级 |

无需额外脚本。加载此 skill 时，agent 自动加载 rubrics-model-evaluation + red-team-verifier，按上述格式输出。

## 财经溯源时间窗口门禁（v1.2.0新增 — 2026-07-03 英伟达翻车教训）

当用户要求溯源财经文章/事件的可靠性时，**强制先执行时间窗口锁定**，不可跳过：

### 时间窗口锁定（Step 0 — 溯源启动前必做）

1. **确定事件时间**：文章发布日期 ± 72h 窗口
2. **搜索一手官方源**：公司博客 / PR Newswire / CFO联署公告 / SEC 8-K filing
3. **搜索独家爆料**：The Information / Bloomberg / Reuters 原创新闻
4. **锁定后**：确认「最近72h有没有新事件」——有→以此为锚；无→扩展到更早分析

**门禁不通过（跳过1-3直接搜分析文）→ 溯源结论不可发布。**

翻车案例（2026-07-03）：分析微信公众号"英伟达的算力贷"时，跳过了NVIDIA 7月1日CFO联署官方博客+The Information 7月1日独家爆料，直接搜到了Tomasz Tunguz几周前的旧分析，导致结论完全相反。

## 系统设计方案评审（v1.2.0）

Rubrics 框架同样适用于系统设计方案评审（如日报差异化方案、架构改造提案）。维度需做适配：

| 原始维度 | 方案评审对应 | 评估要点 |
|------|------|------|
| ① 事实正确性 | 对现状的诊断是否准确 | 是否基于代码/日志/实测数据？ |
| ② 覆盖度 | 是否覆盖了所有关键问题域 | 有无遗漏的维度（如LLM prompt层）？ |
| ③ 证据支撑 | 每条建议是否有根因支撑 | 是否引用了具体数据/日志/用户反馈？ |
| ④ 推理严谨性 | 因果链/参数推导是否清晰 | 阈值有无依据（如"衰减50%"为什么是50%）？ |
| ⑤ 格式合规性 | 方案结构是否完整清晰 | 落地步骤是否有序？ |
| ⑥ 来源类型 → 可维护性 | 改动侵入性/风险 | 是否可能引入新bug？ |
| ⑦ 实际可用性 | 能否直接落地 | 改动量多大？是否需要新依赖？ |
| ⑧ 意义提炼 | 是否超越了表面修补 | 有没有系统性解决根因而非打补丁？ |

|**适用场景**（2026-07-03 实例）：日报差异化方案评审——衰减+轮换+prompt引导，红队识别出衰减参数拍脑袋、去重误杀风险、配额与质量冲突三个疑点。|

## 验收标准评审（v1.6.0新增）

当三合一分析报告的验收标准自检表被标记为「全面评审」触发时，Rubrics 评估维度需额外检查：

| Rubrics 维度 | 验收标准对应 | 评审要点 |
|------|:--:|------|
| ① 事实正确性 | 标准①②③④ | 数字标注是否完整？指标是否实际计算？ |
| ② 覆盖度 | 标准⑥⑧ | 反共识声明是否充分？赛道适用性是否声明？ |
| ③ 证据支撑 | 标准①⑤ | 来源标注率？复核通过率？ |
| ④ 推理严谨性 | 标准⑦ | 仓位推导是否来自一致性矩阵？ |
| ⑤ 格式合规性 | 自检表格式 | 自检表是否逐条对照？ |

## 三模型并行评审 (v1.18.0 · ai-berkshire 启发)

**来源**：xbtlin/ai-berkshire 的 4 Agent 并行分析模式 + 本系统 2026-07-13 落地。

当需要最高质量审查时，不使用单一 delegate_task（同模型串行），而是拆分三个独立 cron，各自绑不同模型同时运行：

| 角色 | 模型 | 架构 | 审查目标 |
|------|------|------|------|
| 🔴 红队数据审计 | `deepseek-v4-flash` | Dense | 数据准确/逻辑自洽/区间可复现 |
| 🔄 反共识压力测试 | `deepseek-v4-pro` | Dense | 发散思维/反事实推演/死因假设 |
| 🌐 宏观映射检验 | `MiniMax-M3` | MoE | 跨域知识/外部一致性/SOX映射 |

**正交性来源**：
- Flash vs Pro：认知模式不同（精确规则匹配 vs 深度推理）
- DeepSeek vs MiniMax：架构不同（dense vs MoE），训练数据不同
- 三个 cron 独立上下文，不共享推理链

**实现**：三个 cron 同 schedule（如 `20 9 * * 1-5`），读同一个 trajectory 文件，各自输出 8-10 行紧凑结论。见 `references/parallel-review-architecture.md`。

**对话场景替代方案**：对话中无法复现三模型并行（delegate_task 不支持 per-task 模型）。两种降级：
1. **对话模式**：主进程内分三段输出（认知多样性保留，模型多样性丢失）
2. 若分析已写入 trajectory → `cronjob run` 触发三个 cron 手动跑

**⛔ 对话场景输出格式规则（v1.20.0）**：在对话中以"三路审查"格式输出时，必须遵守以下免责声明：

- 若为**真实三路并行**（三个 cron 独立触发）：标注「✅ 三模型正交审查已完成」
- 若为**对话模拟**（单模型扮演三角色）：标注「⚠️ 本次三路审查为单模型模拟，未经独立模型/独立上下文验证。若需真正正交审查，可回复「全面评审」触发 cron 并行审查。」
- **禁止**在模拟三路审查的底部标注「本分析未经红队独立审查」——因为格式中已经嵌入了一个名为🔴红队的段落，读者会自然追问"那段红队算什么"。模拟红队≠独立审查员，声明必须在行文顶部就区分清楚。

> 反例（2026-07-14 机器人ETF午间审查）：输出标题为"三路审查"，正文含🔴红队/🔄反共识/🌐宏观映射三段，底部却标注"本分析未经红队独立审查"——用户立即指出自相矛盾。根因：单模型角色扮演的红队分析本质上仍是同模型生成，与真实独立审查有认知同质性差距。

**适用场景**：
- 盘前/盘后的自动化评审（cron 模式）
- 系统设计方案/架构提案的深度评审
- 需要真正正交视角的高风险分析

## 两步法 vs 一步法（v1.3.0新增 · 元模式）

当用户要求对文章/方案同时做「分析建议」和「质量评审」时，**分离两步优于合并一步**。

**根因**：生成和批判共用同一模型时，批判思维会污染生成思维——产出的建议不是因为全面所以收敛，而是因为不敢发散所以没尝试。这正是 LoopCoder 两圈法则在方法论层面的映射。

**推演**：
- 一步法：同时做评审+建议 → 评审结果里不会出现"建议过度扩展"——因为建议是自己刚生成的，大脑无法同时切换到批判模式
- 两步法：先大胆建议 → 红蓝队独立批判 → 收窄——被砍掉的和没被提出的，不是一个量级

**适用场景**：
- 用户发了外部文章要分析 → 先独立分析提建议 → 再触发全面评审
- 用户直接说"全面评审并给方案" → 主动问："先让我分析文章提建议，再走红蓝队评审？"

**翻车案例**：英伟达算力贷溯源——Grace 两步法找到 NVIDIA 7/1 官方博客，隐序一步法搜到旧分析就下结论。时间窗口门禁强制了两步分离。

## 外部文章修正（v1.1.0新增）

当用户发送外部文章链接（如 EarlETF、券商研报、行业分析）时，若文章内容对当前分析有修正作用，应在**整合结论之前**将其纳入：

1. 提取文章中与当前分析直接相关的判断
2. 对照自己的分析，标注哪些被验证、哪些需修正
3. 修正后的结论写入整合总结

**典型场景**（2026-07-02 实例）：科创50短线分析完成后，EarlETF 文章指出「A股半导体无法脱离日韩美股走出独立行情」→ 修正了交易前置条件（增加"等待美股企稳"的门槛）。

## 理解债前置（v1.27.0 · 协作纪律）

**在文章评审类全面评审启动前**，如果当前会话已经累积了多轮对话、多个任务线、或跨 session 的 compacted context，须先执行「理解债清理」：

1. **问自己**：当前任务依赖的前置条件中，有哪些是"我以为我知道但实际可能错的"？
2. **查活文件**：compacted context 是旧快照——SOUL.md / MEMORY / USER.md 可能已在两次会话之间被修改。先读活文件，再读快照。
3. **标未确认**：若有涉及数字/事实/数据源的前置判断未经验证，在全面评审输出中显式标注为"基于未验证前提"。

> 最危险的理解债不是"知道不知道"，而是"不知道自己在用错数据"（如 2026-07-23 科创50 trajectory 的 `close_last=2.016` vs 实际昨收 `1.955`，所有下游分析基于错数运行）。

## Understanding Debt 闸门（v1.14.0 · P2）

> 来源：Addy Osmani / ClaudeDevs Loop Engineering 文章。AI 产出大量"看起来正确但团队不理解"的内容 → 累积"理解债"。

**在所有分析类输出末尾追加此闸门**：

```
## 🏦 理解债闸门

本次输出中，以下关键判断**未经人工确认**：

| # | 判断 | 为什么需要确认 | 确认方式 |
|:--:|------|------|------|
| 1 | {自动生成的判断} | {如果错了会怎样} | {人工复核/交叉验证/等待实盘反馈} |

> 若全部关键判断均已经过人工/实盘验证 → 标注「✅ 本次无未确认关键判断」
```

**规则**：
- 至少列出 1 条。无法识别任何未确认判断 = 最大的理解债风险。
- 判断来源包括：LLM 推理结论、自动计算的指标、自动生成的定档、自动选用的方法论框架
- 不阻塞交付——闸门是"标记"而非"拦截"。目的是让用户意识到"这个结论我没验证过"

> **最危险的理解债（2026-07-23 实战教训）**：不是"知道不知道"，而是"不知道自己在用错数据"。科创50 trajectory 的 `close_last=2.016` 实际昨收 `1.955`，所有下游分析（deviation_20、刘晨明离场阈值、入场区间）基于错数运行了一整天。如果红队没在午盘手动发现，这个错会静默累积到下一周。**闸门不仅要标记"我们不确定的判断"，还要标记"我们以为确定的数字"——后者更难发现，伤害更大。当数据源有多个候选（K线 close vs quote.last_close），且其中一个是隐性错误源（ETF K线返回NAV而非市场价）时，即使判断看起来"确定"，也应该在闸门中追问"这个数字从哪来的"。**
