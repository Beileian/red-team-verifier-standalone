# 跨源验证：DojoAgents 案例 + 操作清单

**日期**: 2026-07-23
**触发**: 用户发送微信文章《一周多狂揽1000+Star！金融Agent开始进入真实工作流》

## 事件

1. 全面评审文章时，基于文章描述对 DojoAgents 做了 Rubrics 评分，结论为 B 级（7.7/10）

5. 深入读 `dojo_sdk_tool.py` 源码 → 发现核心事实：**所有数据来自 HuggingFace 离线数据集**

## 发现

文章描述"调用全球市场工具拉取 A股/美股/港股异动"，在技术上为真——Agent 确实调用了这些工具。但工具读的是深涌智能预处理的 HuggingFace 离线快照，不是实时行情。

14 个工具绑定，每个描述都以 "from offline HuggingFace datasets" 结尾。

## 强制三步验证（操作清单）

当全面评审对象是引用第三方开源项目的文章时，Rubrics 评分前必须执行：

| 步骤 | 检查内容 | 操作 |
|:--:|------|------|
| **Step 1** | Star/维护状态 | 访问 GitHub 首页 → 核实 Star 数、Fork、最后 commit 日期、License |
| **Step 2** | 关键功能声明 | 在源码中搜文章核心宣称的功能关键词（如 `real-time`/`offline`/`batch`），读对应文件 |
| **Step 3** | 数据源/依赖 | 检查 requirements.txt / import 语句 / registry / config → 确认数据是 API 还是离线数据集 |

### 裁决规则

| 文章与源码一致性 | 行动 |
|------|------|
| 高度一致 | Rubrics 正常进行 |
| 重大遗漏 | 红队追加「文章遗漏」疑点，事实正确性降 1 分 |
| 事实相反 | 红队标记为致命疑点，事实正确性降 2 分 |

## 数字确定性闸门（v1.27.0 · close_last 教训）

> 2026-07-23：科创50 trajectory 的 `close_last=2.016` vs 实际昨收 `1.955`，所有下游分析基于错数运行。

**最危险的理解债**：不是"知道不知道"，而是"不知道自己在用错数据"。

当数据源有多个候选，且其中一个来源对特定品种存在隐性偏差（ETF 的 K 线 close 返回 NAV 而非市场价）时，"确定"的判断恰恰是最该被追问的。

### 闸门规则

在全面评审结论中，除标记"我们不确定的判断"外，还要标记至少 1 条「**我们确定的数字**」及其来源：

```
🏦 理解债闸门

| # | 类型 | 判断/数字 | 来源 | 如果错了 |
|:--:|:--:|------|------|------|
| 1 | ⚠️ 不确定 | AI capex 支付能力将在 Q3 见底 | LLM 推理 | 入场偏离至少一个季度 |
| 2 | 🔢 确定但可能错 | deviation_20 = -4.27% | K线末行close | 离场阈值误判，风险敞口翻倍 |
```
