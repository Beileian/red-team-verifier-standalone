# 底盘检查——建管道前提条件

> 来自 2026-07-06 投资线自进化评审教训

## 模式

当提议「在现有数据管道上加一层推送/自进化/评估管道」时，**必须先检查底层数据管道的健康度**，再决定是否建上层建筑。

## 检查清单

```
□ 上游数据源是否正常产出？（cron last_status、文件 mtime）
□ 评估/回填是否按时运行？（state 文件时间戳 vs 预期频率）
□ 数据字段是否完整？（如 outcome 是否为空、lessons 是否生成）
□ 是否存在时序依赖？（上游 cron 在推送 cron 之后运行 → 空数据）
```

## 翻车实例

| 日期 | 提案 | 底盘问题 | 后果 |
|------|------|------|------|
| 2026-07-06 | BTC 周度推送（整合三 eval） | 交易 eval cron error、宏观 eval 2 周未更新、PnL verifier 4 天过期 | 如果直接建推送，用户看到的是过期数据或空报告 |
| 2026-07-06 | deviation_weekly 推送 | 脚本存在但从不产出 state 文件，4 条 trajectory 只有 3 条有 lessons | 推送的周报数据量不足，方向准确率基于 1 个样本 |

## 修正后的流程

```
1. 列出所有上游数据源 → 检查每个的健康度
2. 修复已损坏的数据源（先修底盘）
3. 确认修复后稳定运行 ≥1 个周期
4. 再建上层推送/评估管道
```

## 反例

日报自进化机制的底盘是健康的（生成 cron + eval cron 都正常），所以可以直接加推送。但不能把这个模式未经检查地照搬到投资线。
