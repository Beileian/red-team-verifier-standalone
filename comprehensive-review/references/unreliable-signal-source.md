# 不可信信号源陷阱（unreliable-signal-source）

> 2026-07-09 BTC 投递静默丢包修复方案评审实例

## 背景

BTC 虚拟量化系统 cron（78c08a885914, deliver=origin）的投递出现两种失败模式：
- **7:42**：iLink rate limit → 调度器捕获错误 → 日志有记录
- **13:42**：iLink HTTP 200 → 调度器标记 success → gateway.log 无任何投递记录 → 消息未达微信

## 三个被评审的方案

| 方案 | 核心假设 | 依赖的信号源 |
|------|------|------|
| A. 调度器加送达确认 | `_deliver_result` 返回值可靠 | gateway.log（出站日志） |
| B. 投递看门狗升级 | cron 跑完后 2 分钟内 log 有记录=成功 | gateway.log（出站日志） |
| C. BTC post-hook 补发 | 30 秒轮询 log 确认后补齐 | gateway.log（出站日志） |

## 红队关键发现

**gateway.log 的 send 记录 ≠ 微信聊天窗口收到消息。**

三个失败场景下的对照：

| 失败场景 | gateway.log 状态 | A | B | C |
|---------|-----------------|---|---|---|
| HTTP 根本没发出（13:42） | 无记录 | ❓调度器已误报 | ✅能检测 | ✅能检测 |
| **HTTP 200 但 iLink 丢包** | **有 send 成功记录** | **❌误判成功** | **❌误判成功** | **❌误判成功** |
| 日志延迟落盘 | 暂时无记录 | ❌误判失败 | ❌误判失败 | ❌误判失败→重复消息 |

最危险的场景是第二种——**三个方案全部误判为成功，制造虚假的安全感。** 而这种场景的发生频率可能不低：iLink HTTP 200 只表示接受消息，不保证送达微信客户端。

## 元教训：信号源可信度检查

任何系统设计方案评审时，追加此检查：

1. **方案依赖的核心信号源是否被独立验证过？**
   - gateway.log 的 send 记录是否被端到端测试过（发一条 → 确认收到 → 查 log 一致）？
   - 如果从未验证 → 建立该验证是先决条件，而非依赖它做仲裁

2. **该信号源是否存在"false positive"场景？**
   - 有记录但消息未送达？有记录但被延迟？有记录但送达了错误的会话？
   - 如果 false positive 存在 → 方案必须额外说明如何处理

3. **如果信号源不可信，方案是否仍然有效？**
   - 如果答案是否 → 方案需要重新设计

## 正确方向：从 push 确认到 pull 模式

当面确认不可行时，改产品逻辑比改投递代码更可靠：

> **BTC 报告改 deliver=local → 用户主动拉取。**
> 用户想看的时侯发消息触发推送。这是唯一能消除"是否送达"不确定性的方案——用户主动拉取时，如果通道有问题会自己感知到。
