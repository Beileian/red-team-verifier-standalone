# Cron 窗口冲突分析评审实例

> 2026-07-17 · comprehensive-review 已知陷阱 #21

## 背景

分析 08:30-09:30 工作日窗口内 34 个 cron 的时序冲突、资源争用、成本错配。产出 5 个风险 + P0-P3 优先级排序。用户要求全面评审该分析。

## 红队三个致命疑点

### 1. P0 标签建立在未验证假设上

四条 pipeline（国电/银行/机器人/科创50）的数据采集→审查间隔均为 10 分钟。原始分析将此标为 P0 时序竞态。

**反证**：数据采集脚本是 no_agent Python 脚本，做 HTTP 拉取腾讯 qt + 同花顺 K 线，典型耗时 <30 秒。"10分钟不够"的判断无一条实测数据支撑。唯一引用的"银行宏观空跑"案例根因是 trajectory 缺宏观字段（设计缺陷），非采集超时。

### 2. 成本数字被不成比例放大

6 次 DeepSeek LLM 调用落在高峰 2x 费率，被标为"中危"。实际成本差：6 次 × ¥0.01 = ¥0.06/天 = ¥1.8/月。与"时序竞态""推送风暴"并列为中危稀释了风险等级区分度。

### 3. 维修方案缺验证闭环

四个优先级的建议全部是"改 cron schedule"，零验证步骤。15 个 cron schedule 改动无一条附带 "cronjob run + 检查输出 + 确认下游" 的测试计划。本质上是 7/15 "cron 创建后零测试即部署"陷阱的变体。

## 蓝队方向一：先测后修

用户采纳 P0：创建诊断 cron（每周五 09:35，no_agent），用 `morning_window_diag.py` 收集 34 个 cron 的实际 `last_run_at`、`last_status`、输出文件大小、DeepSeek API 延迟。攒一周实际数据后再决定修复方案。

脚本产出：`~/.hermes/cron/state/morning_window_diag.json`

## 教训

1. **P0 需要数据**：未经验证的假设不能支撑 P0 优先级
2. **成本用绝对数字**：¥0.06/天 直接用金额标注，不套风险等级
3. **维修带验证**：任何 cron schedule 变更必须附带测试步骤
4. **先测后修 > 推测后修**：诊断 cron 收集实测数据驱动决策，优于基于推测的批量改动
