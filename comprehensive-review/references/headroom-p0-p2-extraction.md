# headroom P0→P1→P2 萃取方法论

> 来源：headroomlabs-ai/headroom 全面评审 (2026-07-16, Rubrics 8.0/B级)
> 价值：不是"评审了一个项目"——是"从评审到落地三阶段的完整方法"

## 场景

对一个高评分外部项目（59k⭐）做全面评审后，不是简单地"装或不装"，而是拆成三层递进落地：

| 阶段 | 做什么 | 外部依赖 | 风险 |
|:--:|------|:--:|:--:|
| **P0** | 萃取设计思想 → 零依赖注入本系统 | 无 | 零 |
| **P1** | Library 模式 A/B 测试 | `pip install` | 低（单 cron 试点） |
| **P2** | 架构策略文档化 | 无（改现有 skill） | 低 |

## P0：设计思想萃取（不要装系统，学会它的思路）

headroom 的三个模式映射到 Hermes：

| headroom 模式 | Hermes 映射 | 落地 |
|------|------|------|
| CCR（可逆压缩+检索） | MEMORY 静默截断→显式索引 | 文档化 |
| Live-zone（只压最新不动历史） | MEMORY 增量 patch 替代全量重写 | memory-tiering v1.5.0 |
| Fail-gracefully（失败=原样返回） | 关键路径降级链规范 | output-integrity-gate §23 |

**核心原则**：看到一个好系统，不要问"装不装"——先剥三层看设计模式，挑能用的零成本注入。

## P1：Library 模式 A/B 测试（验证假设，不全量采纳）

- 安装 `pip install headroom-ai`（17.8MB wheel，2min）
- 写 `headroom_ab_test.py`，三个样本基准：web_search JSON / 冗余 JSON / 微信文章文本
- 实测结果：中文财经 JSON 10% 节省、冗余 JSON 57%、纯文本 0%
- 决策：10% 节省 + 零质量损失 → 高热 cron 可试点，但 v0.x API 需监控

**核心原则**：不因为全面评审给高分就全量采纳——拿真实数据跑 A/B，数字说话。

## P2：架构策略文档化（让教训可复用）

- memory-tiering v1.4.0→v1.5.0：Live-zone 增量策略
- engineering-patterns v1.9.0→v1.10.0：三模式文档 + AB 测试数据
- output-integrity-gate v1.13.0→v1.14.0：Fail-gracefully 安全网

**核心原则**：每个阶段的产出都写入 skill，不靠记忆。

## 反模式（不要做的事）

- ❌ 全面评审给 8.0/B 级 → 直接装（跳过了 P0/P1）
- ❌ AB 测试结果好 → 全量推（跳过了"v0.x API 稳定吗"）
- ❌ 模式萃取后只记 MEMORY → 下次遇到同类项目从零开始

## 适用条件

这个三阶段方法适用于：
- 外部项目评分 ≥ 7.0 但存在落地风险（依赖不稳定/中文未验证/资源需求超 VPS 能力）
- 项目有可独立萃取的设计思想（不只是"API 好用"）
- 用户明确要求评估后落地（不是被动参考保留）

不适用于：
- 评分 < 5.0 → 不值得花三阶段
- 纯工具类项目（如 CLI 工具）→ 装或不装，没有"设计思想"可萃取
- 与 Hermes 完全重叠的平台级项目 → 跳过
