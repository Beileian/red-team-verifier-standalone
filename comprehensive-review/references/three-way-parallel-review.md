# 三路并行审查模式 · 2026-07-13

## 来源

xbtlin/ai-berkshire 的 4-Agent 并行分析模式 → 适配为 3 路独立 cron 审查。

## 架构

```
09:10  盘前三合一分析（deepseek-v4-flash）
         ↓ 写入 trajectory JSON
09:20  三路并行审查（同时触发）：
         ├─ dccb36a07524: 红队数据审计    → deepseek-v4-flash（精确+便宜）
         ├─ 99d1b0bf2b7d: 反共识压力测试  → deepseek-v4-pro（推理+发散）
         └─ 61ff16f16be0: 宏观映射检验    → MiniMax-M3（MoE架构正交）
```

## 模型选择依据

| 任务 | 模型 | 架构 | 理由 |
|------|------|------|------|
| 数据审计 | Flash | Dense | 规则匹配型任务——对照清单逐条打勾，不需要深度推理，便宜80% |
| 反共识 | Pro | Dense | 需要真正生成非共识洞见——推理链更长，更可能产出意外发现 |
| 宏观映射 | M3 | MoE | 跨域知识+模式识别——不同架构天然提供最强正交性 |

## 正交性来源

- Task A vs B：**认知模式不同**（精确规则匹配 vs 发散创造）——同架构也能拉开距离
- Task B vs C：**架构不同**（Dense vs MoE）——不同训练数据+不同推理机制
- A vs C：**架构+认知模式双不同**——最大差异化

## 对话场景替代方案

`delegate_task` 不支持 per-task 模型选择 → 对话中无法复现三模型并行。

替代：主进程触发三个独立 cron 的 `action='run'`（需先将分析写入临时 trajectory 文件作中转）。操作：用户说"三路审查"→ 我写文件→触发三个 cron→回收结果。

## 设计决策

C 方案（3 独立 cron × 3 模型）优于 D 方案（观察再决定）的权衡：额外维护成本 3 个 cron prompt vs 收益——真正的架构级正交性（dense vs MoE、精确 vs 发散）。
