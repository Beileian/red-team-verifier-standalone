# 红蓝子代理辩论模式

## 适用场景

当用户要求评估系统设计方案/架构改进提案时，使用 `delegate_task` 启动两个独立子代理进行红蓝对抗。

## 为什么有效

红蓝队共用同一模型时，批判思维会污染生成思维——"一步法"产出的建议不是因为全面所以收敛，而是因为不敢发散所以没尝试。用独立子代理（独立上下文、独立工具栈）做红蓝对抗，本质上是 **LoopCoder 两圈法则在方法论评估层面的映射**。

## 实战案例（2026-07-04）

### 场景 1：科创50改进方案评估

**触发**：用户要求对 P0-P2 四条算法改进方案做红蓝队讨论。

**红队 context**：
- 四条方案的详细描述
- 用户的核心目的（"开盘前对波动区间做预判，交易时段动态触发更好"）
- 指令：不留情面地攻击——失效场景、成本/复杂度、是否掩盖更深层根因

**蓝队 context**：
- 同样四条方案的描述
- 指令：在红队批判基础上建设性完善——保留核心价值、提出更优雅替代方案、优先级调整

**产出**：
- 红队识别出"四大暗线"：①增量补丁无结构性质变 ②过度依赖静态硬阈值 ③解决显示问题非预测问题 ④需求理解偏差
- 蓝队提出了替代方案："开盘情境卡片"→替代P0、"超卖仪表盘"→替代P1·A、"预测快照协议"→替代P2·A、"价格闹钟"→替代P2·B
- 红蓝共识：预测快照协议是MVP基础

### 场景 2：muyun-sibian 方法论评估

**触发**：用户问"让红蓝队辩论一下这个skill的方法论体系和与三合一的整合方式。以及适合前评估还是后评估？"

**红队 context**：
- muyun-sibian v2.0 的三大支柱完整描述
- 当前整合方式："先跑三合一再补充"
- 指令：攻击方法论逻辑缺陷、整合方式确认偏误、前评估vs后评估适用性、可操作性

**蓝队 context**：
- 同样内容
- 指令：在红队批判基础上建设性完善——修正方案、重新设计整合架构、明确前/后评估定位、最小可行方案

**产出**：
- 红队致命攻击5条：三支柱不在同一抽象层、交易心性跑错片场、先三合一再补充=确认偏误、框架不可证伪性、信号到决策断裂
- 蓝队立即执行修正：拆柱（pillar-1→Step 2.4、pillar-2→Step 3.5、pillar-3→独立对话）、版本降级 v0.3.0-beta、7条已知局限

## ⚠️ 已知陷阱：并行 vs 串行的上下文共享

当前 `delegate_task` 的 `tasks` 数组是**并行执行**的。这意味着蓝队子代理**读不到红队子代理的输出**——两者同时启动，彼此上下文隔离。

在 muyun 辩论中，蓝队子代理的 summary 明确写了：
> "Red Team output was delivered directly to WeChat and isn't retrievable from the filesystem"

蓝队通过**自身对系统已知缺陷的理解**弥补了无法读取红队输出的限制——但这不是最优方案。

**正确的顺序**：

```
方案：先红后蓝（串行）
  ① delegate_task → 红队子代理（独立批判）
  ② 主进程读取红队输出 → 作为 context 注入蓝队
  ③ delegate_task → 蓝队子代理（在红队批判基础上建设）

备选：并行 + 主进程整合（当前方案）
  ① delegate_task tasks=[红, 蓝] → 并行
  ② 主进程读取两队输出 → 自做整合结论
  ③ 优点：更快；缺点：蓝队无法引用红队的具体攻击点
```

**选择建议**：
- 需要蓝队**逐一回应**红队攻击点 → 用串行
- 只需要两队**独立视角**供主进程综合 → 用并行
- 当前 muyun 辩论用了并行 + 蓝队自主推测红队论点 → 勉强可用但不够精确

## ⚠️ 已知陷阱 2：子代理修改文件后父进程必须重新读取

当蓝队子代理在辩论中直接修改 skill 文件（如 muyun-sibian 和 three-in-one-analysis 的 SKILL.md），父进程**不会自动感知**这些变更。2026-07-04 实战中：

- 蓝队子代理修改了 `muyun-sibian/SKILL.md`（v2.0.0→v0.3.0-beta）和 `three-in-one-analysis/SKILL.md`（v1.4.0→v1.5.0）
- 父进程通过 `find -newer` 检测到文件变更，通过 `grep version:` 发现了版本号变化
- 如果不主动检测，父进程可能基于旧版本继续操作，造成版本混乱

**预防规则**：
1. 红蓝辩论结束后，父进程**必须**用 `find -newer` 检查所有相关 skill 文件是否被修改
2. 如有修改，`read_file` 重新读取完整内容后再决定下一步
3. 将子代理的文件修改视为"已执行的修正"，父进程只做验证和确认，不做覆盖式修改
