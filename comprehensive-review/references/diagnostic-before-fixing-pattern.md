# 先测后修：诊断驱动的 cron 优化模式

> 2026-07-17 · 来自 cron 窗口审计评审蓝队方向一

## 模式

当面对"N 个 cron 可能存在时序冲突/资源争用"的分析时，不基于推测直接改 schedule。先创建一个诊断 cron 收集实际运行数据，用数据驱动修复决策。

## 三步法

### Step 1: 创建诊断脚本

no_agent cron，在目标窗口结束后触发，收集每个目标 cron 的：
- `last_run_at`（实际触发时间 vs 排程时间）
- `last_status`（成功/失败）
- 输出文件大小和时间戳（验证产出完整性）
- API 延迟（如适用）

### Step 2: 等待数据积累

至少跑一个完整周期（一周/一天，取决于窗口频率）。不急于修。

### Step 3: 数据驱动修复

用 JSON 输出识别：
- 真正超时的 cron（`last_run_at` vs schedule 偏差 > 阈值）
- 真正失败的 cron（`last_status != ok`）
- 输出为空的 cron（`total_bytes = 0`）

只修复被数据证实的冲突。不被"理论上可能"的冲突驱动改代码。

## 与"推测后修"的对比

| | 推测后修 | 先测后修 |
|------|------|------|
| 输入 | 时间线分析 + 假设 | 实测数据 |
| 改动量 | 可能 15 个 cron | 通常 2-3 个 |
| 回滚风险 | 高（改多了） | 低（只改需要改的） |
| 引入新冲突 | 可能 | 可控 |
| 时间成本 | 立即动手 | 等一周数据 |
